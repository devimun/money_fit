# 앱 확장성을 고려한 ID 전략: AUTOINCREMENT에서 UUID로

로컬 우선(Offline-First)으로 앱을 개발할 때 많은 개발자가 자연스럽게 `AUTOINCREMENT`를 기본 키(Primary Key)로 사용합니다. 저 또한 그랬습니다. 하지만 이 결정이 미래의 확장성에 어떤 영향을 미치는지, 그리고 왜 UUID로 전환하게 되었는지 그 과정을 공유하고자 합니다.

### 1. 초기 설계의 함정: `AUTOINCREMENT`

사이드 프로젝트 'MoneyFit'을 처음 설계할 때, 모든 데이터는 사용자의 디바이스 내에만 저장되는 것을 전제로 했습니다. SQLite를 사용했고, 지출 내역 테이블의 스키마는 다음과 같았습니다.

```sql
CREATE TABLE expenses (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    amount REAL NOT NULL
);
```

이 방식은 로컬 환경에서는 매우 직관적이고 효율적입니다. 하지만 프로젝트가 발전하면서 '서버를 통한 데이터 동기화'라는 요구사항이 생겼을 때, 이 설계는 근본적인 문제에 부딪혔습니다.

- **사용자 A의 디바이스:** `id`가 `1, 2, 3`인 지출 내역 생성
- **사용자 B의 디바이스:** `id`가 `1, 2, 3`인 지출 내역 생성

두 사용자의 데이터를 하나의 서버 데이터베이스에 통합하려 할 때, `id` 값이 필연적으로 충돌합니다. 이 문제를 해결하려면 서버에서 ID를 발급받거나, 클라이언트에서 ID를 생성할 때 복잡한 오프셋(offset) 전략을 사용해야 합니다. 이는 오프라인 상태에서의 데이터 생성을 어렵게 만들고 시스템의 복잡도를 높입니다.

### 2. 문제 해결의 열쇠: UUID (Universally Unique Identifier)

이 문제에 대한 표준적인 해결책은 UUID입니다. UUID는 서버와의 통신 없이도 클라이언트에서 독립적으로 거의 100%에 가까운 고유성을 보장하는 ID를 생성할 수 있게 해줍니다.

UUID(버전 4 기준)는 128비트의 숫자 중 122비트를 무작위 값으로 채워 생성됩니다. 경우의 수가 2^122 (약 5.3 x 10^36)에 달해, 전 세계 모든 기기에서 동시에 UUID를 생성해도 충돌할 확률은 실질적으로 0에 수렴합니다.

이러한 특성을 활용하기 위해 데이터베이스 스키마를 다음과 같이 수정했습니다.

```sql
CREATE TABLE expenses (
    id TEXT PRIMARY KEY, -- UUID 저장을 위해 TEXT 타입으로 변경
    user_id TEXT NOT NULL,
    name TEXT NOT NULL,
    amount REAL NOT NULL
);
```

이제 클라이언트는 `uuid`와 같은 라이브러리를 사용해 직접 ID를 생성하고, 이 ID를 기본 키로 사용하여 데이터를 저장할 수 있습니다. 서버는 클라이언트가 생성한 ID를 그대로 신뢰하고 사용하기만 하면 됩니다.

### 3. 추가 고려사항: 충돌 확률과 엔지니어링 트레이드오프

"그래도 이론적으로 충돌할 가능성이 있지 않은가?"라는 질문이 생길 수 있습니다. 맞습니다. 확률은 0이 아닙니다.

만약 충돌이 발생하면 데이터베이스의 `PRIMARY KEY` 제약 조건에 의해 `INSERT`가 실패할 것입니다. 이 경우, `try-catch` 문으로 에러를 감지하고 새로운 UUID로 재시도하는 방어 로직을 구현할 수 있습니다.

하지만 여기서 엔지니어링 트레이드오프에 대한 고려가 필요합니다. UUID의 충돌 확률은 무시할 수 있을 정도로 낮습니다. 이 희박한 가능성을 처리하기 위해 코드를 추가하고 테스트하는 비용보다, 실제로 빈번하게 발생하는 네트워크 오류나 데이터 유효성 검증 로직에 집중하는 것이 훨씬 더 효율적입니다. 따라서 UUID 충돌은 발생하지 않는다고 가정하고, 관련 예외 처리 로직은 구현하지 않는 것으로 결정했습니다.

### 4. 최종 아키텍처 설계

UUID를 ID 전략으로 채택하면서, 백엔드 아키텍처에 대한 선택지도 훨씬 유연해졌습니다.

- **인증 및 데이터베이스:** 관계형 데이터 모델의 장점을 살리기 위해 PostgreSQL 기반의 **Supabase**를 메인 백엔드로 선택했습니다. Supabase Auth를 통해 사용자를 관리하고, 모든 데이터는 Supabase DB에 저장합니다.
- **부가 서비스:** 사용자 행동 분석(Analytics)이나 광고 수익화(AdMob)처럼 Google 생태계가 강점인 기능들은 **Firebase** 서비스를 연동하여 사용하기로 했습니다.
- **익명 로그인 구현:** Supabase는 Firebase와 같은 직접적인 익명 로그인 기능을 제공하지 않습니다. 대신, 앱 최초 실행 시 UUID 기반의 가짜 이메일로 `supabase.auth.signUp()`을 호출하여 '익명 사용자'와 동일한 경험을 제공합니다. 사용자가 나중에 소셜 로그인을 선택하면, `updateUser()`를 통해 계정 정보를 업데이트합니다. 이 과정에서 사용자의 고유 `UID`는 변하지 않으므로 데이터 연속성이 완벽하게 보장됩니다.

### 정리 및 요약

`AUTOINCREMENT`에서 UUID로의 전환은 단순한 데이터 타입 변경 이상의 의미를 가집니다.

1.  **확장성 확보:** 서버 동기화 및 분산 환경�� 고려한 확장 가능한 구조의 기반을 마련했습니다.
2.  **오프라인 우선 지원:** 서버 의존성 없이 클라이언트에서 ID를 생성함으로써, 오프라인 상태에서도 모든 기능이 원활하게 동작합니다.
3.  **유연한 아키텍처:** 각 서비스의 장점(Supabase의 RDB, Firebase의 부가 기능)을 선택적으로 조합하는 하이브리드 아키텍처 구성이 가능해졌습니다.

사이드 프로젝트를 진행하며 겪은 이 경험은 확장성을 고려한 초기 설계의 중요성을 다시 한번 깨닫게 해주었습니다. 만약 지금 새로운 프로젝트를 시작한다면, 미래를 위해 ID 전략을 한번쯤 깊이 고민해 보시길 추천합니다.
